# Memory Consolidation - 记忆整理与压缩

## 🎯 定位

模拟睡眠巩固记忆：短期日志 → 结构化长期记忆。按项目分区，置信度驱动。

> **详细参考**（索引结构、分区规则、置信度体系、Search Before Building、Workflow闭环、Escalation规则、报告模板）→ `references/detailed-protocols.md`

---

## 📋 六阶段流程

| Phase | 名称 | 核心动作 | 产出 |
|-------|------|---------|------|
| **1** | Orient | 读 MEMORY.md + 扫描日志目录 + 确定范围 | 整理范围 |
| **2** | Gather | 读日志 → 提取偏好/决策/经验/踩坑 | 关键信息 |
| **3** | Consolidate | 按主题归类 → 分配置信度 → 写入分区 → 合并重复 | 更新后的 MEMORY.md |
| **4** | Prune | 删除过期信息 → 归档 30 天+ 日志 | 整理报告 |
| **5** | Rebuild Index | 扫描 MEMORY.md ## 标题 → 全量重建 memory-index.json | 更新的索引 |
| **6** | Audit Trail | 追加执行摘要到 memory-index.json 的 audit_history | 审计记录 |

> Phase 1-4 详细步骤 + Phase 5 索引重建流程 → 见 references §1-3

## 🔑 关键约束

### 置信度（必须标注）

| 分数 | 含义 | 触发 |
|------|------|------|
| [0.3] | 试探性 | 提及 1 次 |
| [0.5] | 有倾向 | 2-3 次或选择但未强调 |
| [0.7] | 偏好确认 | 明确偏好或多次遵循 |
| [0.9] | 核心习惯 | 明确说"记住"或 5+ 次确认 |

晋升：[0.3]→[0.5] 第 2 次 → [0.5]→[0.7] 第 4 次 → [0.7]→[0.9] 用户确认。降级：30 天未引用降一级。

### 健康指标

| 指标 | 健康 | 行动 |
|------|------|------|
| MEMORY.md 大小 | **≤ 4000 chars** | 超限强制裁剪（借鉴 Hermes Agent 容量限制设计） |
| 未处理日志天数 | < 7 天 | > 7 天需整理 |
| 日志文件总数 | < 30 个 | > 30 个需归档 |

### 容量硬限制（v4.0 新增）

**MEMORY.md 上限：4000 chars**（中文信息密度约为英文 2 倍，Hermes 限制 2200 chars 英文）

| 超限程度 | 处理方式 |
|---------|---------|
| > 4000 chars | **Phase 3 裁剪优先级：** ① [0.3] 试探性条目 ② >60 天未引用 ③ 操作步骤类（应归 Skill） ④ 冗余描述 |
| > 6000 chars | 立即触发整理，不等定期维护 |
| 超限写入被拒 | 先删除/压缩旧条目，再写入新条目（与 Hermes `MemoryStore` 同策略） |

**核心原则**：低质量 Memory 注入上下文 = 每次 API 调用都带噪声。宁可少记，不可滥记。

### 保留 vs 丢弃

- ✅ 保留：用户说"记住"、重复模式（≥2次）、技术选型、里程碑
- ❌ 丢弃：中间过程、临时搜索、重复日常操作、30天未引用条目、琐碎交互
- 🔄 迁移：操作步骤/踩坑经验 → 归入对应 Skill 的踩坑区（Memory 只留声明式事实）

## ⚙️ 核心规则

1. **最小权限**：默认不删除，用户确认后才删
2. **保留原文**：归档是移动而非重写
3. **透明操作**：每次整理生成变更报告
4. **不过度概括**：提炼时保留足够细节
5. **项目隔离**：按工作区路径匹配加载，不跨项目污染
6. **置信度驱动**：所有偏好条目必须带 [0.3]-[0.9] 前缀
