# 方法论：从关键词匹配到语义路由的演进

## 核心问题

AI Agent 系统中，Skill 路由本质上是一个**意图分类问题**：

```
输入: 自然语言 Query
输出: 最适合处理该 Query 的 Skill ID
```

这个问题看似简单，工程上却有很多坑。

---

## 第一代：关键词匹配

```python
for trigger in skill.triggers:
    if trigger in query:
        return skill
```

**结果：66.7% 准确率**

**问题根源：**
- 用户说"帮我拉一下利润表" → 触发词里没有"利润表"和"拉一下"
- 用户说"再看看它的财务" → "再"、"它的"完全不在关键词库里
- 维护者无法预知所有可能的表达方式

---

## 第二代：TF-IDF + 中文分词

关键洞察：**与其穷举关键词，不如让系统自己计算相似度**。

```python
# Skill 的"代表向量" = 其描述文本的 TF-IDF 表示
# Query 的向量 = 用户输入的 TF-IDF 表示
# 路由 = 找余弦相似度最高的 Skill
```

中文分词策略：
- 不引入 jieba（减少依赖）
- 用 bigram + trigram 滑动窗口代替词典分词
- 效果：`"利润表"` → `["利", "润", "表", "利润", "润表", "利润表"]`

**结果：83.3% 准确率**

---

## 第三代：类别先验 + 优先级仲裁

问题：金融类 Skill 多（neodata / finance-data / stock-analyst / dcf / comps / lbo...），向量相似度经常在这几个里打平。

**解法：**
1. 检测到金融领域词 → 金融类 Skill 分数 × 1.5
2. 同类 Skill 分数差 < 15% → 按业务优先级规则决胜

```python
CATEGORY_PRIORITY = {
    "金融数据": ["neodata-financial-search", "finance-data-retrieval"],
    # neodata 优先于 finance-data，因为前者更通用
}
```

**结果：91.7% 准确率**

---

## 第四代：精确词快速通道

剩余失败案例分析：全是专业缩写词（DCF、LBO、PRD、CIM）。

**根本原因**：这些词本身就唯一确定意图，不需要语义计算。

**解法**：正则预匹配层，命中即直接返回，置信度 1.0。

```python
EXACT_OVERRIDE = {
    r'\bdcf\b': "dcf-model",
    r'\blbo\b': "lbo-model",
}
```

**结果：100% 准确率（12/12）**

---

## 关键设计原则

### 1. 分层而非单一策略

不要试图用一个算法解决所有情况。精确词、语义词、上下文词，各有各的最优处理方式。

### 2. 快速通道 + 兜底链

```
精确词（100% 准确，快）→ 语义向量（~85% 准确，中）→ LLM rerank（~95%，慢）→ fallback
```

只有前一层失败才走下一层，保证平均延迟最小。

### 3. 上下文是免费的信息

对话历史里隐含着大量有用信息（实体、已用工具、任务线索），注入 Query 只需几十行代码，收益巨大。

### 4. 在线学习比离线训练更实用

对 Skill 路由这类规模的问题（几十个类别），在线收集纠错数据、实时更新触发词，比定期重训模型更敏捷。

### 5. 置信度门控 > 强制选择

宁可返回 "confirm"（低置信，询问用户），也不强行路由到错误 Skill。错误路由的代价远高于多一次确认。

---

## 推广思考

这套方法论不仅适用于 WorkBuddy，对任何"意图 → 工具"的路由场景都有参考价值：

- ChatBot 的多 FAQ 路由
- 企业内部知识库的文档检索
- 自动化工作流的任务分发
- 多模型 API 的智能路由（GPT vs Claude vs Gemini）

**核心抽象**：把"工具选择"变成"语义检索"，把"关键词维护"变成"描述文本优化"。
