# Self-Improvement 详细参考

> **何时读此文件**：需要执行完整自我优化流程、生成报告、创建 Skill 候选、或执行跨会话模式检测时查阅。
> SKILL.md 保留了核心维度、步骤摘要和关键规则。

## 1. 意图预判层（详细版）

### 完整预判流程

```
收到用户请求
    ↓
扫描所有 Skill 的 name + description（仅标题行，不加载全文）
    ↓
判断：是否有 ≥ 1 个 Skill 能显著提升任务效率？
    │
    ├── 有 → 预加载该 Skill（即使触发词未精确命中）
    │       预加载优先级：
    │       1. 任务类型精确匹配（如金融数据 → finance-data-retrieval）
    │       2. 领域相关匹配（如股票分析 → stock-analyst）
    │       3. 工具链匹配（如写代码 → code-review 准备联动）
    │
    └── 无 → 正常执行，不做多余加载
    ↓
执行过程中如果发现效率低下 → 中途补充加载 Skill
```

### 预判示例表

| 用户说 | 触发词匹配 | 意图预判 | 应该加载 |
|--------|-----------|---------|---------|
| "帮我拉一下利润表" | 可能没命中 | 金融数据检索 | `finance-data-retrieval` |
| "这个代码帮我看看" | 可能没命中 | 代码审查 | `code-review` |
| "推送到 GitHub" | `git-workflow` | Git 操作 | `git-workflow` |
| "做个调研" | `web-research` | 研究任务 | `web-research` + `Deep Research` |
| "帮我记一下" | 可能没命中 | 知识管理 | `腾讯ima` 或记忆写入 |

### 反模式

- ❌ 用户说"你好"→ 不预判加载任何 Skill
- ❌ 简单的文件读取请求 → 不预判加载重型 Skill
- ❌ 不确定的领域 → 先执行，观察后再决定

## 2. 质量评分系统

### 评分维度

| 维度 | 权重 | 评分标准（1-5 分） |
|------|------|-------------------|
| **目标达成** | 30% | 1=未完成 3=部分完成 5=完全达成 |
| **效率** | 25% | 1=严重超支 3=符合预算 5=优于预算 |
| **工具选择** | 15% | 1=多次选错 3=基本正确 5=最优选择 |
| **用户理解** | 15% | 1=多次误解 3=基本理解 5=精准理解 |
| **知识复用** | 15% | 1=重复造轮子 3=部分复用 5=充分利用已有 Skill/记忆 |

### 计算方式

```
总分 = Σ(维度分 × 权重) → 1.0 - 5.0

质量等级：
  ≥ 4.5 → ⭐⭐⭐⭐⭐ 优秀（可作为 Skill 创建的输入）
  ≥ 3.5 → ⭐⭐⭐⭐ 良好（正常水平）
  ≥ 2.5 → ⭐⭐⭐ 合格（有改进空间）
  < 2.5 → ⭐⭐ 需反思（必须触发深度改进分析）
```

### 评分持久化格式

将评分写入 memory-index.json 的 `quality_history`：

```json
{
  "quality_history": [
    {
      "date": "2026-04-25",
      "session_type": "功能开发",
      "scores": {
        "goal": 5, "efficiency": 4, "tools": 4, "understanding": 5, "reuse": 3
      },
      "total": 4.35,
      "grade": "⭐⭐⭐⭐",
      "key_improvement": "可以更多利用子智能体卸载探索性工作"
    }
  ]
}
```

### 趋势分析规则

| 模式 | 含义 | 行动 |
|------|------|------|
| 连续 3 次 ≥ 4.0 | 稳定高质量 | 当前 Skill/工作流有效，维持 |
| 连续 2 次下降 ≥ 0.5 | 质量下滑 | 触发深度反思：是什么变了？ |
| 特定维度持续 ≤ 3 | 系统性短板 | 针对该维度创建改进 Skill 或更新规则 |
| 总分波动大（极差 ≥ 2） | 不稳定 | 可能与任务类型/上下文压力相关，分析关联 |

## 4. 自动触发决策流程（详细版）

### 触发条件总表

| 触发场景 | 优化重点 | 最低阈值 |
|---------|---------|---------|
| 会话中出现 3 次以上重试 | 工具选择 + 踩坑经验记录 | 同一错误 ≥ 2 次 |
| 用户明确表达不满意 | 用户理解 + 沟通方式改进 | 任何负面反馈 |
| 完成一个复杂多步骤任务 | 工作流效率 + Skill 封装机会 | ≥ 5 个工具调用 |
| 用户分享了新的偏好或约束 | 立即写入 MEMORY.md | 任何新偏好 |
| 任务完成后自动 Skill 候选识别 | 从执行轨迹提取可复用模式 | 任务含 ≥ 3 个可复用步骤 |
| 跨会话模式识别 | 与 MEMORY.md 对比，发现重复模式 | 同类任务 ≥ 2 次会话出现 |

### 决策流程图

```
任务/会话中的一个里程碑点
    ↓
检查触发条件（按优先级）
    │
    ├── 1. 用户明确说"反思/复盘/总结"
    │       → 立即执行完整反思流程（Step 0-5 + 质量评分）
    │
    ├── 2. 用户表达不满（"不对""重新来""不是这样的"）
    │       → 立即执行维度 2（用户理解）专项反思
    │       → 标记本次会话质量评分权重调整（用户理解维度权重临时 +50%）
    │
    ├── 3. 工具调用失败重试 ≥ 2 次（同一操作）
    │       → 记录踩坑经验到对应 Skill
    │       → 不执行完整反思（避免打断流程），在任务完成后一并反思
    │
    ├── 4. 任务阶段完成（工具调用累计 ≥ 5 次）
    │       → 快速自检：回顾已执行步骤，是否有冗余/可优化
    │       → 如果发现明显低效，立即调整后续策略
    │       → 不中断用户流程，静默优化
    │
    ├── 5. 任务全部完成
    │       → 执行完整反思流程（Step 1-4）
    │       → Skill 候选识别（如果 ≥ 3 个可复用步骤）
    │       → 质量评分（Step 5）
    │       → 写入日志 + 更新记忆
    │
    └── 6. 用户分享了新偏好/约束
            → 立即写入 MEMORY.md（带置信度）
            → 不执行完整反思
```

### 工具调用计数规则

| ✅ 算 1 次 | ❌ 不计数 |
|------------|----------|
| read_file, write_to_file, replace_in_file, search_content, search_file, execute_command, list_dir | read_lints（诊断检查） |
| use_skill（加载 Skill） | conversation_search（上下文检索） |
| web_search, web_fetch | update_memory（记忆更新） |
| task（子智能体派发） | todo_write（任务管理） |
| RAG_search | deliver_attachments（结果交付） |

> 统计时机：从会话第一个工具调用开始累计，每任务完成后重置计数器。

## 5. Skill 自动创建 MVP 流程

### 创建条件（必须满足 ≥ 2 条）

| 条件 | 说明 | 示例 |
|------|------|------|
| 步骤可复用 | ≥ 3 个有序步骤，可被不同输入重复执行 | "读取配置→验证格式→写入数据库" |
| 领域专精 | 涉及特定领域知识、API 模式或工作流 | 保险产品定价流程、量化回测模板 |
| 跨会话重复 | 类似任务在 ≥ 2 次会话中出现 | "每次都要手动查 API 文档" |
| 现有 Skill 未覆盖 | 检查 Skill 注册表中无匹配项 | 没有任何 Skill 处理"合同审查" |

### 创建流程

```
任务完成 → 触发 Skill 候选识别
    ↓
满足创建条件？
    ├── 否 → 跳过
    └── 是 → 提取模式 → 生成模板 → 用户审核 → 写入 → 注册
```

### 标准 SKILL.md 模板

```markdown
---
name: {skill-name}

description: >
  {一句话描述功能}。

complexity: ⭐⭐

  - {使用的工具列表}
---

# {Skill 名称}

## 🎯 技能定位
{详细的功能描述}

当用户输入某种表述但本 Skill 未被自动激活时，完成任务后**必须**执行：
1. 分析用户原始请求中的关键表述
2. 将其抽象为通用触发词
3. 用 replace_in_file 追加到 YAML frontmatter description 末尾
4. 不得重复添加已存在的触发词
```

## 6. 跨会话模式检测引擎

### 检测流程

```
新会话开始
    ↓
Step 1: 读取 memory-index.json（极低成本）
    ↓
Step 2: 匹配用户请求关键词与 cross_session_patterns
    ↓
Step 3: 判断结果
    │
    ├── 高匹配（≥ 2 个 pattern 命中，或频率 ≥ 5）
    │   → 主动告知用户 + 预加载对应 Skill
    │
    ├── 中匹配（1 个 pattern 命中，频率 2-4）
    │   → 静默预加载对应 Skill
    │
    └── 无匹配 → 正常执行
```

### 模式记录 JSON 格式

```json
{
  "pattern": "量化数据查询",
  "frequency": 8,
  "last_seen": "2026-04-24",
  "suggested_skill": "finance-data-retrieval",
  "key_lessons": ["ts_code 必须带后缀", "频率参数格式有限制"]
}
```

### 字段维护规则

| 字段 | 说明 | 更新时机 |
|------|------|---------|
| `pattern` | 模式名称 | 首次识别时创建 |
| `frequency` | 出现次数 | 每次匹配时 +1 |
| `last_seen` | 最后出现日期 | 每次匹配时更新 |
| `suggested_skill` | 推荐的 Skill | 模式稳定后设定 |
| `key_lessons` | 关键经验摘要 | 从踩坑经验中提取 |

### 模式衰减规则

- 30 天未出现：frequency 减半
- 60 天未出现：标记为 `[休眠]`，不再主动推荐
- 90 天未出现：从索引中移除

## 7. D5 序列模式检测（v3.0 新增）

> 借鉴 Hermes Agent 安全审计的"双重视角"——不仅逐条看每步是否合理，还要串起来看是否构成低效/重复模式。

### 检测方法

```python
# 不是真代码，是反思时的思维框架
def detect_sequence_pattern(tool_call_history):
    patterns = []

    # 1. 低效循环检测
    if detect_cycle(tool_call_history, min_length=3):
        patterns.append("低效循环：相同操作序列重复 ≥ 3 次")

    # 2. 工具惯性检测
    dominant_tool = most_used_tool(tool_call_history)
    if dominant_tool.ratio > 0.7 and has_alternative(dominant_tool):
        patterns.append(f"工具惯性：{dominant_tool.name} 占比 >70%，可能存在更优选择")

    # 3. 重复任务检测（跨会话）
    if session_pattern.frequency >= 2:
        patterns.append(f"重复任务：'{session_pattern.name}' 在 {session_pattern.frequency} 次会话中出现")

    return patterns
```

### D5 评估清单

| 检查项 | 示例 |
|--------|------|
| 这 N 步是否构成循环？ | 读文件→修改→报错→再读→再改→又报错 |
| 工具选择是否有惯性？ | 总是用 search_content，其实直接 read_file 更快 |
| 是否在重复已有 Skill 的工作？ | 手动做了 Skill 已覆盖的流程 |
| 是否有"假效率"（快但错）？ | 快速给出答案但方向错了 |

### D5 驱动的改进动作

| 发现的模式 | 改进动作 |
|-----------|---------|
| 低效循环 | 更新对应 Skill 的 Escalation 规则 |
| 工具惯性 | 记入踩坑经验，下次优先尝试替代工具 |
| 重复任务 | 触发 Skill 候选创建（见 §7 扩展触发） |
| 假效率 | 在 D1 评分中标记效率扣分 |

## 8. Skill 候选扩展触发条件（v3.0 更新）

### 创建条件（满足任一即触发，v3.0 更新后以此为准）

> **注意**：§5 的旧规则"满足 ≥ 2 条"已被本节覆盖。

| 条件 | 说明 | 示例 |
|------|------|------|
| 步骤可复用 | ≥ 3 个有序步骤 | "读取配置→验证格式→写入数据库" |
| 领域专精 | 涉及特定领域知识/API | 保险产品定价流程 |
| **跨会话重复**（新增） | **类似任务在 ≥ 2 次会话中出现** | "每次都要手动查 API 文档" |
| **D5 模式识别**（新增） | **D5 反思识别出重复操作序列** | "反复搜索→整理→输出，应自动化" |
| 现有 Skill 未覆盖 | 注册表中无匹配项 | 没有任何 Skill 处理"合同审查" |

> 原规则要求"满足 ≥ 2 条"，v3.0 放宽为"满足任一"——宁可多建一个候选让用户审核，也不要漏掉。

## 9. 效率审计 Addendum（v3.0 新增）

> 借鉴 Hermes Agent 安全审计 Prompt（SECURITY_ADDENDUM），为 self-improvement 增加效率审计视角。

### 在 Step 2 反思时追加的审计视角

```
--- Efficiency Audit ---

额外审视：从"效率分析师"视角重新审视本次会话。

1. 工具调用序列中，是否存在"串行等待但本可并行"的步骤？
2. 是否有步骤的结果未被后续步骤使用（无效计算）？
3. 是否在已经知道答案的情况下仍然执行了验证操作（过度确认）？
4. 如果重新执行这个任务，最少需要几步？（与实际步数对比）

将发现写入 D5 序列模式区。如果存在 ≥ 2 个效率问题，在改进承诺中必须包含对应的优化计划。
```

### 效率评分追加规则

在 Step 5 质量评分的"效率"维度中，额外检查：
- 并行机会利用率：本可并行但串行执行的步骤数 / 总步骤数
- 无效步骤率：结果未被使用的步骤数 / 总步骤数
- 实际步数 vs 最优步数比：> 1.5 倍时效率维度降 1 分
