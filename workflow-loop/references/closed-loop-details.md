# 工作流闭环详细流程与模板

> **何时读此文件**：需要执行具体闭环时，查阅对应闭环的完整流程图、步骤模板和输出格式。
> SKILL.md 中只保留了每个闭环的摘要。

---

## 闭环1：🐛 Bug 修复（详细版）

### 完整流程图

```
用户报告 Bug
     ↓
┌─────────────────────────────────────────┐
│  Step 1: Search Before Building         │
│  Layer 1: 项目内搜索类似问题             │
│  Layer 2: find-skills 查找解决方案      │
│  Layer 3: 第一性原理分析根因            │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 2: 根因调查 (investigate)         │
│  - 重现问题                             │
│  - 追踪数据流                           │
│  - 定位问题代码                         │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 3: 修复实施                       │
│  - 实施修复                             │
│  - 编写/更新测试防止回归                │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 4: 验证 + 提交                   │
│  - 验证修复有效                         │
│  - Git commit + 描述                    │
└─────────────────────────────────────────┘
     ↓
✅ DONE
```

### Step 1 · Search Before Building（强制）

```bash
# Layer 1 - 项目内搜索
grep -r "错误信息\|错误类型" --include="*.log" --include="*.md" .
cat .workbuddy/memory/*.md | grep -i "bug\|问题"

# Layer 2 - 查找现成方案
use_skill: find-skills
# 关键词：debug, investigate, error handling

# Layer 3 - 第一性原理
# 问题出在哪里？数据流追踪？
```

### Step 2 · 根因分析模板

```markdown
## 根因分析

**问题描述**：[用户看到的问题]
**期望行为**：[应该怎样]
**实际行为**：[实际怎样]

**根因**：[找到的根本原因]

**修复方案**：[最简单的修复方式]
```

### Step 3 · Escalation Rules

| 触发条件 | 处理 |
|---------|------|
| 3次尝试无法重现 | STOP，标记"无法重现"，要求更多信息 |
| 涉及多个模块 | STOP，拆分问题 |
| 需要改架构 | STOP，升级为功能开发闭环 |

### Step 4 · 输出格式

```markdown
## Bug修复报告

STATUS: DONE

**问题**：{简要描述}
**根因**：{找到的原因}
**修复**：{做了什么}
**验证**：{如何确认修复有效}
**测试**：{添加了什么测试}
```

## 闭环2：🚀 功能开发（详细版）

### 完整流程图

```
需求提出
     ↓
┌─────────────────────────────────────────┐
│  Step 1: Search Before Building         │
│  Layer 1: 项目内是否有类似功能          │
│  Layer 2: find-skills 查找实现方案      │
│  Layer 3: 设计新方案                   │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 2: 需求确认                       │
│  - 理解真正需求（不只是表面描述）       │
│  - 明确边界条件                         │
│  - 确定验收标准                         │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 3: 实现 + 自测                    │
│  - TDD 或边写边测                       │
│  - 写注释和文档                         │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 4: Review + 修复                  │
│  use_skill: code-review                 │
│  - 修复发现的问题                       │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 5: 提交 + 文档                    │
│  - Git commit                           │
│  - 更新相关文档                         │
└─────────────────────────────────────────┘
     ↓
✅ DONE
```

### Step 1 · Search Before Building（强制）

```bash
# Layer 1 - 项目内搜索
grep -r "功能关键词" --include="*.py" --include="*.ts" --include="*.js" .
grep -r "TODO\|feature\|TODO" --include="*.md" .

# Layer 2 - 查找现成方案
use_skill: find-skills
# 关键词：implementation, feature development

# Layer 3 - 第一性原理设计
# 如果没有现成方案，设计最简单的实现
```

### Step 2 · 需求确认模板

```markdown
## 需求确认

**原始需求**：[用户说的]
**真正需求**：[理解后的本质]
**非需求**：[明确不包括的范围]
**验收标准**：[如何判断完成]
```

### Step 3 · Escalation Rules

| 触发条件 | 处理 |
|---------|------|
| 需求模糊不清 | STOP，要求澄清 |
| 涉及3+模块改动 | STOP，拆分成多个 PR |
| 需要外部依赖 | STOP，确认依赖可行性 |

### Step 4 · 输出格式

```markdown
## 功能开发报告

STATUS: DONE

**功能**：[开发了什么]
**实现**：[关键设计决策]
**测试**：[写了什么测试，覆盖什么场景]
**文档**：[更新了哪些文档]
**下一步**：[可能的扩展方向]
```

## 闭环3：📖 研究调研（详细版）

### 完整流程图

```
研究问题
     ↓
┌─────────────────────────────────────────┐
│  Step 1: Search Before Building         │
│  Layer 1: 知识库/内存中是否已有         │
│  Layer 2: find-skills 查找相关 skill   │
│  Layer 3: 设计研究策略                 │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 2: 执行研究                        │
│  use_skill: web-research               │
│  - 多角度搜索                           │
│  - 交叉验证                             │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 3: 整理输出                        │
│  - 结构化报告                           │
│  - 持久化到知识库（如需要）             │
└─────────────────────────────────────────┘
     ↓
✅ DONE
```

### Step 1 · Search Before Building（强制）

```bash
# Layer 1 - 知识库搜索
cat .workbuddy/memory/*.md | grep -i "研究关键词"
ls .workbuddy/knowledge/ 2>/dev/null | grep -i "关键词"

# Layer 2 - 查找相关 skill
use_skill: find-skills
# 关键词：research, analysis, investigation

# Layer 3 - 设计研究策略
# 哪些问题必须回答？
# 需要哪些信息源？
```

### Step 2 · 研究问题模板

```markdown
## 研究问题

**研究主题**：[要研究什么]
**核心问题**：[需要回答的具体问题]
**研究边界**：[不包括的范围]
**交付形式**：[报告/对比表/决策建议]
```

### Step 3 · Escalation Rules

| 触发条件 | 处理 |
|---------|------|
| 5+次搜索无有效信息 | 标记"信息不足"，输出已知部分 |
| 发现矛盾信息 | 呈现各方观点，标注置信度 |
| 涉及敏感内容 | STOP，明确拒绝 |

### Step 4 · 输出格式

```markdown
## 研究报告

STATUS: DONE

**研究主题**：[主题]
**核心发现**：[最重要的3-5点]
**置信度**：[高/中/低]
**来源**：[信息源列表]
**下一步**：[建议的下一步行动]
```

## 闭环4：🔍 代码审查（详细版）

### 完整流程图

```
审查请求
     ↓
┌─────────────────────────────────────────┐
│  Step 1: Search Before Building         │
│  Layer 1: 检查代码规范文档              │
│  Layer 2: find-skills 查找审查工具      │
│  Layer 3: 基于最佳实践审查              │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 2: 执行审查                        │
│  use_skill: code-review                │
│  - 安全 → 质量 → 性能 → 可维护性 → 测试  │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 3: 生成报告                        │
│  - 按严重级别分类                         │
│  - 提供可执行的修复建议                   │
└─────────────────────────────────────────┘
     ↓
┌─────────────────────────────────────────┐
│  Step 4: 跟进                            │
│  - 跟踪修复进度（如需要）                │
│  - 验证修复有效性                        │
└─────────────────────────────────────────┘
     ↓
✅ DONE
```

### Step 1 · Search Before Building（强制）

```bash
# Layer 1 - 项目规范搜索
cat .eslintrc .pylintrc rustfmt.toml 2>/dev/null || echo "无规范文件"
grep -r "规范\|convention" --include="*.md" .

# Layer 2 - 查找审查工具
use_skill: find-skills
# 关键词：code review, linting, security

# Layer 3 - 基于最佳实践
# OWASP / CERT / CWE 安全标准
```

### Step 2 · 审查重点模板

```markdown
## 审查重点

**审查范围**：[哪些文件/模块]
**审查维度**：
- 🔴 安全（最高优先级）
- 🟠 代码质量
- 🟡 性能
- 🔵 可维护性
- 🟢 测试覆盖
```

### Step 3 · Escalation Rules

| 触发条件 | 处理 |
|---------|------|
| 发现 Blocker 级安全问题 | 立即报告，不继续 |
| 同一问题出现 3+ 次 | 建议系统性修复 |
| 涉及认证/加密 | STOP，要求人工复核 |

### Step 4 · 输出格式

```markdown
## 代码审查报告

STATUS: DONE | DONE_WITH_CONCERNS

**范围**：[审查的文件]
**总问题数**：🔴 N | 🟠 N | 🟡 N | 🟢 N

**优先行动项**：
- [ ] 🔴 立即修复：{Blocker 列表}
- [ ] 🟠 本周修复：{High 列表}
- [ ] 🟡 下次迭代：{Medium 列表}
```
