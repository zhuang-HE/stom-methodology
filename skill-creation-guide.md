# SKILL.md 编写与优化方法论 — STOM v2.1

> **STOM** = Skill Token Optimization Method（Skill Token 优化方法论）
> **目标**：控制每个 Skill 加载时的 token 消耗，同时不丢失必要信息。
> **核心理念**：SKILL.md 是"执行指令手册"，不是"完整参考文档"。每个 token 都要有存在的理由。

---

## 一、硬性指标

| 指标 | 上限 | 说明 |
|------|------|------|
| **SKILL.md 主文件行数** | **≤ 350 行** | 超出内容必须移至 `references/` |
| **SKILL.md 主文件大小** | **≤ 15 KB** | 约 300-500 tokens/KB（含中文时更高） |
| **YAML frontmatter** | ≤ 25 行 | 只放元数据 + description 触发词 |
| **references/ 子文件** | 不限 | 按需 read_file，不自动加载 |

---

## 二、三层信息架构（核心模型）

```
┌─────────────────────────────────────────────────┐
│  L1 执行层 — SKILL.md（常驻上下文）              │
│  → AI 开始执行任务时"必须知道"的信息              │
│  → 限制：≤350行 / ≤15KB                         │
├─────────────────────────────────────────────────┤
│  L2 参考层 — references/（按需加载）              │
│  → AI 遇到复杂场景时"需要查阅"的信息             │
│  → 扁平结构，禁止互相引用                        │
├─────────────────────────────────────────────────┤
│  L3 外部层 — 实时搜索（不存储）                  │
│  → 写代码/调试时才需要的完整文档                 │
│  → 通过 search / web_search / API 文档获取       │
└─────────────────────────────────────────────────┘
```

### L1 vs L2 vs L3 决策矩阵

| 判断维度 | L1 SKILL.md | L2 references/ | L3 外部 |
|---------|------------|---------------|--------|
| **使用频率** | 每 task 都会用到 | 部分 task 用到 | 偶尔用到 |
| **使用时机** | 开始执行前就需要 | 执行中途遇到时 | 写代码/调试时 |
| **丢失后果** | 任务无法正确完成 | 质量降级但不失败 | 可通过搜索恢复 |
| **典型内容** | 接口速查表、参数格式、校验规则 | 工作流模板、示例、详细错误处理 | 完整 API 文档、SDK 源码 |
| **大小约束** | ≤350行 / ≤15KB | 单文件 ≤200KB | 无限制 |

### 内容频率分布验证法

新建或审计 Skill 时，用此矩阵决定每块内容的归宿：

```
                    高频(每task)   中频(30-70%)    低频(<30%)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
核心执行指令         → L1            → L1           → L2
详细流程/模板        → L2            → L2           → L2  
示例/启发           → L2            → L2           → 移除
完整文档/FAQ         → L3            → L2           → L3
```

**只有落在「高频 + 核心执行」格子的内容才有资格留在 L1。**

## 三、文件结构模板

```
skill-name/
├── SKILL.md                    # 主文件（≤350行）：执行时必须看的内容
├── _meta.json                  # 元数据（如有）
├── references/                 # 详细参考（按需读取，扁平结构）
│   ├── workflow-templates.md   # 完整工作流/步骤模板
│   ├── examples.md             # 典型示例
│   ├── detailed-rules.md       # 扩展规则（质量/缓存/错误处理等）
│   └── *.md                    # 其他参考资料
└── scripts/                    # 可执行脚本（如有）
```

### 🔴 架构铁律：禁止引用链

```
✅ 允许：SKILL.md → references/workflow-templates.md     （一层引用）
❌ 禁止：workflow-templates.md → "详见 detailed-rules.md"  （二层引用）

原因：多层引用导致 AI 需要 N 次 read_file，抵消剥离的 token 节省。
      如果 ref 文件中确实需要某条规则知识，内联一句话概括即可。
```

## 四、内容分级规则

### 🟢 必须留在 L1（SKILL.md）的内容

| 类型 | 说明 | 典型行数 |
|------|------|---------|
| YAML frontmatter | name/version/description/credentials/references列表 | 10-25 行 |
| 核心定位（1-2段）| 这个 Skill 干什么、不干什么 | 5-15 行 |
| When to use / NOT for | 触发场景精简列表（表格优先）| 15-30 行 |
| 环境前置校验 | 前置条件 checklist | 5-15 行 |
| 核心 API/接口速查表 | 意图→接口映射（表格形式）| 20-40 行 |
| 关键参数格式 | 必须记住的参数规则 | 10-20 行 |
| 输入/输出规范 checklist | 核心规则，不含详细解释 | 15-25 行 |
| 最佳实践精华 | 3-7 条核心原则 | 5-10 行 |
| 协作说明 | 与其他 Skill 的配合方式 | 10-20 行 |
| 踩坑经验 | 已积累的实际经验条目 | 5-15 行 |
| **复杂任务触发器** | 明确什么条件下必须读 references/ | 5-10 行 |

### 🔴 必须移到 L2（references/）的内容

| 类型 | 存放文件 | 原因 |
|------|---------|------|
| **详细工作流模板**（每步都有说明的流程）| `references/workflow-templates.md` | 占行最多（100-200+行），且只在规划阶段需要 |
| **大量示例列表** | `references/examples.md` | 示例是"启发用"，不是"执行必须" |
| **详细规则扩展**（数据质量清单、缓存策略、错误分层等）| `references/detailed-rules.md` | 出问题时才查，正常执行不需要 |
| **完整的 API 参数文档** | `references/api-doc.md` | 写代码时才需查阅 |
| **FAQ / 常见问题** | `references/faq.md` | 遇到问题才看 |

### 🟡 可留可移（视情况判断）

| 类型 | 判断标准 |
|------|---------|
| Intent taxonomy / 分类体系 | 如果 ≤30行且为表格形式 → 保留；如果每类都有详细说明 → 移到 references |
| Entity resolution rules | 如果 ≤15行 → 保留；更长 → 精简或移出 |
| Error handling | 保留标准话术模板（≤10行），详细诊断矩阵 → 移出 |

## 五、编写风格规范

### 1. 表格优于段落

```markdown
# ❌ 差：段落式（占行多，信息密度低）

### 1. 行情趋势
当用户想查看某只股票最近的走势时，可以使用以下接口：
daily 接口用于获取日线行情数据，pro_bar 接口可以获取前复权...
典型问题包括：最近走势怎么样、今年涨了多少...

# ✅ 好：表格式（信息密度高，一目了然）

| # | 意图 | 核心接口 | 辅助接口 |
|---|------|---------|---------|
| 1 | 行情/趋势 | daily, pro_bar | weekly, daily_basic |
```

### 2. Checklist 优于散文

```markdown
# ❌ 差
在请求数据之前，你需要先做一些准备工作。首先检查 Python 是否可用，
然后确认 tushare 包是否已经安装...

# ✅ 好
## 环境前置校验
- [ ] Python 3.7+ 可用？
- [ ] tushare 包已安装？
- [ ] TUSHARE_TOKEN 存在？
```

### 3. 引用指针代替内联展开

```markdown
# ❌ 差：在主文件中展开所有细节

## 工作流模板
### 1. 单标的行情分析
适用：
- 看下 XX 最近怎么样
... (150 行详解)

# ✅ 好：引用指针 + 一句话概括

## 复杂任务检测（必须读 references/ 的条件）

以下任务执行前 **必须** 先 `read_file references/workflow-templates.md`：

| 任务特征 | 对应模板 |
|---------|---------|
| 用户说"综合简报"/"研究一下"/"全景分析"| 模板 #9 Investment Brief |
| 涉及 5+ 数据源的组合查询 | 模板 #8 或 #9 |
| 用户明确要求"按标准流程" | 所有模板 |

常规查询直接用上方意图→接口映射表执行，无需读取 references/。
```

### 4. 条件触发式指令替代模糊提示

```markdown
# ❌ 模糊提示（AI 容易跳过）
详细工作流模板 → `references/workflow-templates.md`

# ✅ 条件触发式明确规则（AI 必须遵守）
## 复杂任务检测
当任务匹配以下特征时，**必须**先 read_file references/workflow-templates.md：
（上方表格...）
不在上述范围内的任务 → 直接用 SKILL.md 内信息执行，不要读 references/。
```

## 六、剥离风险与 safeguard

### 6.1 剥离后的真实风险

| # | 风险 | 严重程度 | 说明 |
|---|------|---------|------|
| **R1** | AI "忘记"去读 references/ | ⚠️⚠️⚠️ 高 | 最核心风险——引用指针只是文本提示，不是强制调用 |
| **R2** | 多一次工具调用延迟 | ✅ 低 | 多 ~2-3s + ~500 tokens，但净节省更多 |
| **R3** | 引用链导致多次读取 | ⚠️⚠️ 中 | 已通过「禁止引用链」规则消除 |
| **R4** | SKILL.md 与 references/ 版本漂移 | ⚠️⚠️ 中 | 修改接口名时可能漏同步 |

### 6.2 Safeguard 措施

#### ✓ 对策 R1：80-20 原则验证

**SKILL.md 骨架必须能独立完成 ≥80% 的常见任务**，不能把"必需知识"移出去。

检验方法：列出该 Skill 的 Top 10 常见任务逐项检查：
- 仅凭 SKILL.md 信息能否正确执行？→ ✅ 合格
- 需要读 references/ 才能完成？→ ⚠️ 可接受（但应该是低频任务）
- 即使读了 references/ 也不够？→ ❌ 设计缺陷

#### ✓ 对策 R3/R4：架构约束（已在第三节定义）

- 禁止引用链
- 版本号变更约定：
  - `v1.x.y → v1.x.(y+1)`: 只改了踩坑经验或触发词（ref 不受影响）
  - `v1.x.y → v1.(x+1).0`: 结构性变更（**必须检查** ref 同步性）

## 七、四阶段实施流程

### 阶段 0：诊断（现有 Skill 审计）

```
输入：任意 SKILL.md
输出：诊断报告 + 优化方案

① 统计：行数 / KB / 各 section 行数占比
② 分类：将每个 section 标记为 L1/L2/L3
③ 识别问题：
   - L1 中是否有 ≥50行的"参考型"内容？（应移至 L2）
   - L1 中是否有可压缩为表格的段落？
   - L1 中是否有重复信息？
④ 量化预估：优化后行数和节省比例
⑤ 输出审计报告
```

### 阶段 1：设计（新建 Skill）

```
输入：Skill 需求描述
输出：SKILL.md 骨架 + references/ 规划

① 列出所有需要覆盖的场景（意图清单）
② 对每个场景问："AI 执行这个任务时的最小必要知识集是什么？"
③ 最小集合 → L1（SKILL.md）
④ "遇到问题时才查的" → L2（references/）
⑤ "写代码时要查 API 文档的" → L3（不存，实时搜索）
⑥ 用 Content-Frequency Matrix 验证分配合理性
```

### 阶段 2：实施（重构）

```
① 创建 references/ 目录结构
② 按 L2 分类创建子文件（扁平，无互相引用）
③ 在 SKILL.md 对应位置插入引用指针
④ 插入「复杂任务检测」条件触发区块
⑤ 在 YAML frontmatter 的 references 列表中声明
⑥ 更新 version（minor +1）
⑦ 验证：行数 ≤ 350？功能完整性测试？
```

### 阶段 3：持续 safeguard（防止退化）

```
□ 每次 Skill 变更时自检行数上限
□ 新装 Skill 时先审计大小再决定是否安装
□ 定期全量审计（建议每季度）
□ 审计结果记录到本文档末尾「已审计 Skill 记录」表
```

```
□ 触发词是否都在 description 字段中？

```

## 十一、执行模式维度（建议性指导 vs 确定性框架）

根据 `non-motor-insurance-product` 技能改造经验，Skill 可分为两种执行模式，审计时应作为关键考察项：

### 模式对比

| 维度 | 建议性指导 (Advisory Mode) | 确定性框架 (Deterministic Framework) |
|------|---------------------------|-------------------------------------|
| **核心特点** | 提供原则、示例、启发，AI 有自由裁量权 | **强制使用预设脚本/流程**，AI 必须按规范执行 |
| **适用场景** | 创意/探索性任务、开放性场景 | 标准化流程、合规要求、需要一致输出的任务 |
| **风险控制** | AI 可能偏离预期、重新造轮子、输出不一致 | 结果可靠，但灵活性较低 |
| **典型案例** | tushare-data、stock-analyst | non-motor-insurance-product（v2.0） |
| **L1 必备内容** | 核心原则、触发场景、最佳实践 | **强制规则、脚本调用命令、参数格式** |
| **L2 必备内容** | 详细模板、示例、常见错误 | 脚本源码、验证器、标准模板文件 |
| **关键文件** | `references/workflow-templates.md` | `scripts/workflow_controller.py`<br>`scripts/validators.py`<br>`templates/word_format.json` |
| **状态管理** | 通常无状态 | 支持中断恢复（JSON 状态文件） |
| **验证机制** | 建议性检查 | **强制验证**（每步输出格式检查） |

### 设计建议

1. **建议性指导 Skill** → 使用「复杂任务检测」条件触发式引用
2. **确定性框架 Skill** → 必须包含：
   - `scripts/` 目录下的统一入口脚本
   - 输入/输出验证器
   - 标准化模板文件
   - 状态持久化机制
   - SKILL.md 中明确的**禁止重新编写**规则

> **non-motor-insurance-product v2.0 范例**：通过 `workflow_controller.py` + `validators.py` + `word_format.json` 实现确定性执行，确保 AI 只能按预设流程操作。

## 附：Perplexity Skills 设计方法论（STOM v2.1 补充）

> **来源**：Perplexity 官方博客 *Designing, Refining, and Maintaining Agent Skills at Perplexity*
> **定位**：与 STOM v2.1 互补——STOM 管「结构分层」，Perplexity 管「如何写、如何迭代、如何积累」。

### A1. Zen of Skills（与 Python 之禅对照）

| Python之禅 | Skill设计原则 |
|-----------|-------------|
| 简单优于复杂 | Skill是文件夹，复杂性本身就是feature |
| 显式优于隐式 | 激活靠隐式模式匹配，靠渐进披露 |
| 稀疏优于密集 | 上下文昂贵，每个token都要承载最大信号 |
| 特殊情况不足以特殊对待 | Gotcha就是特殊情况，是最高价值内容 |
| 可解释的实现就是好主意 | 如果内容很容易解释清楚，说明模型已掌握，应删掉 |

### A2. Description 是路由触发器

**Description 是 Skill 设计中最难写的一行**：

```yaml
# ✅ 正确写法（描述触发条件）
description: >
  Load when: 用户要求分析某只股票的技术面、问K线形态、缠论买卖点。
  不加载：用户只想查实时行情、财报数据等数据获取需求。

# ❌ 错误写法（描述功能）
description: >
  提供股票技术分析、K线形态识别、缠论买卖点、信号共振评分...
```

必须以 `Load when...` 开头，控制在50词以内，描述用户的真实意图而非Skill的工作流。

### A3. Gotchas 飞轮（Skill 的核心资产）

Gotcha = agent容易翻车的特殊场景/边界情况。这是Skill长期迭代的核心资产，每次翻车追加一条：

```markdown
# references/gotchas.md
```

**Gotcha 追加规则**：
| Agent表现 | 操作 |
|---------|------|
| 任务失败 | 追加一条gotcha |
| 加载了不该加载的Skill | 收紧description + 增加负样本 |

**维护原则**：不要随意修改description（决定路由），大部分时候仅需追加gotcha。

### A4. Skill撰写五步法

1. **先写评估集(Evals)**：真实用户查询 + 已知失败用例 + 邻域混淆样本
2. **写description**：以`Load when...`开头，50词以内，描述用户真实意图
3. **写SKILL.md正文**：给模型留灵活空间，最高价值是gotcha
4. **用好目录结构**：`scripts/`放确定性逻辑，`references/`放重型文档
5. **迭代**：在分支上反复跑评估集

### A5. 六个可落地 Takeaway

1. **Skill不是新文档**——别把README当Skill写
2. **Description是最难的一行**——它决定路由，不是描述，必须以`Load when...`开头
3. **Gotcha是无价的**——出错就加一条，长期飞轮
4. **每个Skill都是税**——加之前先问"agent没它会不会出错"
5. **多模型评测**——别只跟一个模型耦合，同一Skill需在GPT/Claude等不同模型上验证
6. **Action at a distance是真实存在的**——新加一个Skill可能让另一个不相关的Skill变差

### A6. STOM + Perplexity 关系对照

| 维度 | STOM v2.1 | Perplexity方法论 |
|------|-----------|----------------|
| 核心关注 | 信息分层（SKILL.md ≤ 350行） | Skill怎么写、怎么迭代 |
| Description | 要求简洁 | 要求是路由触发器（`Load when...`） |
| 踩坑积累 | 踩坑经验写在SKILL.md末尾 | 独立`references/gotchas.md` |
| 迭代方式 | 定期审计 | 失败用例驱动 |
| 共同点 | 上下文Token意识 | 上下文Token意识 |

**互补关系**：STOM 管「结构」，Perplexity 管「内容与迭代」。
