## 1. Escalation Rules（升级机制）

### 必须立即 STOP 的情况

| 触发条件 | 处理方式 |
|----------|---------|
| 用户正在等待关键决策结果 | STOP，等待决策后再考虑压缩 |
| 压缩可能导致文件编辑冲突 | STOP，确认所有编辑完成后再操作 |
| 连续 3 次压缩均失败或被用户拒绝 | STOP，报告原因并换方案 |
| 用户要求保留的信息无法结构化 | STOP，建议保留完整上下文 |

### Escalation 报告格式

```markdown
STATUS: BLOCKED | NEEDS_CONTEXT
REASON: [问题描述]
CURRENT_PHASE: [研究/规划/实现/测试/交付]
ATTEMPTED: [尝试了什么]
RECOMMENDATION: [建议用户做什么]
```

## 2. Completion Protocol（完成状态协议）

| 状态 | 含义 | 典型场景 |
|------|------|---------|
| **DONE** | 压缩成功，关键信息完整保留 | 正常阶段转换压缩 |
| **DONE_WITH_CONCERNS** | 成功但有风险 | 实现中途被迫压缩 / 部分信息可能丢失 |
| **BLOCKED** | 无法执行 | 用户拒绝 / 时机不当 / 无法结构化 |
| **NEEDS_CONTEXT** | 缺少依据 | 需要用户明确指定要保留什么 |

### 压缩完成报告模板

```markdown
## 上下文压缩报告

STATUS: DONE

**压缩时机**：{阶段/触发条件}
**压缩前**：约 {N} 轮对话
**压缩后**：约 {N} 轮对话  
**压缩率**：{X}%

### 保留
- {项目1}
- {项目2}

### 丢弃
- {项目1}
- {项目2}

**建议**：{是否需要进一步压缩}
```

## 3. Search Before Building（三层检查）

### Layer 1 — 项目内搜索

检查是否有已有的压缩记录和上下文：
- `.workbuddy/memory/*.md` 中的 compact 相关条目
- 今日日志的关键上下文（最近 50 行）
- 需保留的关键文件路径

### Layer 2 — 现有方案查找

- strategic-compact 是否已有该场景的压缩模板？
- workflow-loop 是否有对应的标准压缩指令？
- 其他 skill 有无可复用的压缩模式？

### Layer 3 — 无现成方案时

- 根据当前任务阶段确定优先级
- 设计"保留/丢弃"清单
- 确认时机合适（非编码中途）

## 4. 任务预算估算表（完整版）

| 任务类型 | 预估调用 | 预算(tokens) | 典型场景 | 风险点 |
|---------|---------|-------------|---------|--------|
| 简单查询 | 1-3 | 3K | 查数据/读文件/问答 | 低 |
| 中等任务 | 4-8 | 10K | 改文件/小功能/格式转换 | Skill 加载可能超预期 |
| 复杂任务 | 9-15 | 20K | 新功能/多文件重构/深度调研 | 多 Skill + 大文件读取 |
| 超大型 | 16+ | 30K | 全栈开发/架构设计/系统集成 | 上下文管理是核心风险 |

### 效率评分公式

```
效率 = 任务完成度(0-1) / 工具调用次数
健康阈值: ≥ 0.7
预警阈值: < 0.5 (说明在低效路径上)
```

### 成本优化策略详解

| 策略 | 典型节省 | 适用场景 |
|------|---------|---------|
| 精准读取 (limit/offset) | 20-60% 文件读取成本 | 大文件已知目标区域 |
| 精准搜索 (精确 pattern) | 40-70% 搜索成本 | 知道要找什么时 |
| 避免重复读同文件 | 100% 重复成本 | 最常见的浪费来源 |
| 并行合并无依赖调用 | 减少轮次 | 多个独立操作时 |
| 子智能体卸载 | 极大（主上下文几乎不增）| 探索 20+ 文件时 |
| 快速失败 (3次换路) | 避免无限试错循环 | 方法不确定时 |

## 5. 压缩指令的"保留/丢弃"决策框架

### 判断一个信息块是否该保留

```
问自己三个问题：

1. 后续步骤是否依赖这个信息？
   → 是（路径/函数名/变量名/API参数）→ ✅ 保留
   → 否（搜索过程/试错记录/中间输出）→ ❌ 丢弃

2. 这个信息能否从其他地方恢复？
   → 能（文件还在磁盘上）→ ❌ 可丢弃，需要时重读
   → 不能（用户口说的约束条件/已删除的内容）→ ✅ 必须保留

3. 重建这个信息的成本多大？
   → 低（重新读一次文件）→ ❌ 丢弃
   → 高（需要重新跑一遍分析）→ ⚠️ 视上下文余量决定
```

### 按信息类型的保留优先级

| 优先级 | 信息类型 | 示例 | 原因 |
|--------|---------|------|------|
| P0 必须 | 文件路径 | 当前编辑的文件绝对路径 | 无法反推 |
| P0 必须 | 函数签名/类名 | 已定义但未完成的函数 | 重构的核心锚点 |
| P0 必须 | 变量名 | 关键变量及其含义 | 代码一致性的基础 |
| P0 必须 | 用户约束 | "必须用 Python"、"不能用第三方库" | 用户不重复说 |
| P1 推荐 | 已完成修改列表 | 改了哪些文件、改了什么 | 避免重复工作 |
| P1 推荐 | 决策及依据 | 选了 A 方案因为... | 防止反复摇摆 |
| P2 可选 | 调试结论 | bug 原因是 xxx | 丢失可重新发现 |
| P3 丢弃 | 搜索过程 | 读过的 20 个文件列表 | 可重新搜索 |
| P3 丢弃 | 试错过程 | 第1/2/3次失败的尝试 | 结论已在调试结论中 |
