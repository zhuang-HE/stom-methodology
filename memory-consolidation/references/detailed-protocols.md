## §2 MEMORY.md 分区结构

MEMORY.md 采用 **项目隔离分区** 设计，防止跨项目知识污染：

```markdown
# MEMORY.md

## 🔧 通用偏好（跨项目有效）
- [0.9] 语言：中文为主，技术术语保留英文
- [0.9] 风格：简洁直接，不要冗余寒暄
- [0.7] 输出：优先 Markdown 表格

## 📁 项目：{项目名}
- **技术栈**：{语言/框架}
- **状态**：进行中 / 已完成 / 暂停
- **关键约束**：{项目特定约束}
- **编码偏好**：{项目特定编码风格}
- **踩坑经验**：{项目特定经验}

## 📁 项目：{另一个项目}
- ...
```

### 分区规则

| 规则 | 说明 |
|------|------|
| **通用偏好** | 用户明确声明的跨项目偏好，始终加载 |
| **项目分区** | 按工作区路径或用户声明的项目名分组，仅加载当前项目相关 |
| **新增项目** | 当日志中出现 3 次以上同一项目相关条目时，自动创建新分区 |
| **合并判断** | 如果两个项目名相似度 > 80%（如"保险系统"和"保险产品"），提示用户确认是否合并 |

### 读取策略（v2.1 按需检索）

1. **始终加载**：通用偏好区 + 当前项目分区
2. **按需加载**：通过工作区路径匹配当前项目分区
3. **不加载**：其他项目分区（减少无关信息干扰）
4. **关键词预检**：会话开始时，先扫描 MEMORY.md 的标题行和关键词，仅加载与当前任务相关的分区
5. **记忆摘要**：当 MEMORY.md > 50KB 时，触发摘要压缩；将 [0.3] 置信度的试探性条目压缩为一行摘要

## §3 记忆索引系统（v2.2）

### 索引文件位置

```
.workbuddy/memory/memory-index.json
```

### 索引结构

```json
{
  "version": "1.0",
  "last_updated": "2026-04-24",
  "memory_file_size_kb": 15,
  "sections": [
    {
      "id": "general-prefs",
      "title": "通用偏好",
      "line_start": 8,
      "line_end": 18,
      "keywords": ["语言", "中文", "沟通", "输出", "格式", "置信度", "代码质量"],
      "always_load": true
    },
    {
      "id": "project-quant",
      "title": "项目：量化交易系统",
      "line_start": 21,
      "line_end": 37,
      "keywords": ["Python", "Pandas", "Streamlit", "Tushare", "AkShare", "量化"],
      "always_load": false,
      "workspace_match": ["quant"]
    }
  ],
  "cross_session_patterns": [
    {
      "pattern": "量化数据查询",
      "frequency": 8,
      "last_seen": "2026-04-24",
      "suggested_skill": "finance-data-retrieval"
    }
  ]
}
```

### 索引使用流程

```
会话开始 → 读取 memory-index.json（~1KB）→ 匹配关键词 → 选择分区
  ├── always_load: true → 始终加载
  ├── workspace_match 命中 → 加载
  ├── keywords ≥ 2 个命中 → 加载
  └── 其他 → 跳过
  → 用 read_file(path, offset, limit) 精准读取
```

### 索引维护规则

| 触发时机 | 维护动作 |
|---------|---------|
| 每次更新 MEMORY.md 后 | 同步更新对应分区的 line_start/line_end |
| 新增项目分区时 | 在索引中添加新 section |
| 删除/压缩条目时 | 更新索引的 line 范围 |
| 跨会话模式变化时 | 更新 cross_session_patterns |
| memory-consolidation 整理后 | 全量重建索引 |

### 索引的局限

- 这是**关键词匹配**，不是真正的语义检索（向量嵌入）
- 如果用户的表述与索引关键词不重叠，可能漏加载
- 补救：执行中发现需要额外记忆时，中途补充读取

## §4 置信度评估规则

| 分数 | 含义 | 触发条件 |
|------|------|---------|
| **[0.3]** | 试探性 | 用户提及 1 次，未明确要求记住 |
| **[0.5]** | 有倾向 | 用户提及 2-3 次，或主动选择但未强调 |
| **[0.7]** | 偏好确认 | 用户明确表达偏好，或多次遵循同一模式 |
| **[0.9]** | 核心习惯 | 用户明确说"记住这个"，或 5+ 次重复确认 |

**晋升规则**：[0.3]→[0.5] 第 2 次，[0.5]→[0.7] 第 4 次，[0.7]→[0.9] 用户确认。

**降级规则**：30 天未引用 → 降一级；60 天 → `[已过期]`。

## §6 保留 vs 丢弃原则

### ✅ 应该保留（声明式事实）

借鉴 Hermes Agent："Write memories as declarative facts, not instructions to yourself."

- `'User prefers concise responses'` ✓
- `'Always respond concisely'` ✗
- 用户明确说"记住这个"、"记下来"的内容
- 重复出现（≥2次）的偏好或模式
- 影响未来决策的技术选型（结论，非步骤）
- 项目重要里程碑和关键决策

### 🔄 应迁移到 Skill（操作步骤类）

**v4.0 新增**：Memory 与 Skill 严格分工——Memory 存事实，Skill 存步骤。

| Memory 中常见的错误写法 | 应迁移到 |
|------------------------|---------|
| "Streamlit 中不要用 st.markdown 渲染 HTML，用原生组件更可靠" | → Skill 踩坑区 |
| "ts_code 必须带交易所后缀（000001.SZ），不能只传数字代码" | → Skill 踩坑区 |
| "V10 版本对标 Wind/聚宽/米筐：个股五维分析(雷达图+5维度Tab)..." | → Skill description |
| "表单交互用 selectbox 预设 + text_input 自定义组合模式" | → Skill 工作流步骤 |
| "UI 风格：深色背景(#0a0e17) + 紧凑行距 + hover 蓝色边框" | → Skill 约束区 |

**迁移规则**：下次 memory-consolidation 整理时，扫描 MEMORY.md 中的操作步骤，迁移到对应 Skill。

### ❌ 应该丢弃
- 一次性任务的中间过程细节
- 临时搜索结果（非结论性的）
- 重复的日常操作记录
- 超过30天且未被引用的日志条目
- 过于琐碎的交互（如"帮我改个标点"）
- 超过60天未引用的 [0.3] 条目

## §8 Search Before Building（详细版）

**Layer 1 - 项目内搜索**：
```bash
cat .workbuddy/memory/MEMORY.md 2>/dev/null | wc -l
ls -la .workbuddy/memory/*.md 2>/dev/null | wc -l
grep -i "整理时间|最后更新" .workbuddy/memory/MEMORY.md 2>/dev/null | tail -5
```

**Layer 2 - 查找现有方案**：
- 检查是否有现有的记忆分类模板
- 检查 skill-evolution 是否有相关建议
- 查看其他 skill 是否有需要沉淀到记忆中的经验

**Layer 3 - 如无现有方案**：
- 确定本次整理的优先级（日志太多优先归档，MEMORY.md太大优先精简）
- 设计新的分区结构（如有必要）
- 确定需要保留的核心条目

## §9 Workflow Loop（工作流闭环）

### 上游触发

| 上游场景 | 触发时机 |
|---------|---------|
| 周期性整理 | 建议每周一次，或 MEMORY.md > 100KB |
| 用户要求 | 用户说"整理记忆"、"清理日志" |
| 日志文件过多 | memory/ 目录下 > 30 个日志文件 |
| strategic-compact 触发 | 上下文压缩后需要整理压缩产生的结论 |

### 下游联动

| 下游 skill | 联动方式 |
|-----------|---------|
| `self-improvement` | 记录记忆整理的踩坑经验 |
| `workflow-loop` | 如果需要启动新工作流，提供上下文 |
| `skill-evolution` | 如果发现 skill 需要更新，触发 skill-evolution |

### 记忆整理标准闭环
```
定时/手动触发 → memory-consolidation 扫描 → 提取精华 → 更新 MEMORY.md
                    ↓                              ↓
              归档日志文件                  self-improvement 自检
                    ↓                              ↓
              生成整理报告 ──────────────→ skill-evolution（可选）
```

### 定期维护路径
```
用户会话 → self-improvement 自检 → 发现记忆问题 → memory-consolidation
                    ↓                              ↓
              skill-evolution ←── 多次发现同类问题 ──┘
```

## §10 Escalation Rules（升级机制）

**必须立即 STOP 并升级的情况**：

| 触发条件 | 处理方式 |
|---------|---------|
| 发现疑似敏感/隐私信息 | STOP，报告位置，要求用户确认处理方式 |
| 日志文件损坏或无法读取 | STOP，跳过该文件，记录问题 |
| 删除操作未获用户确认 | STOP，默认只归档不删除 |
| 3次整理尝试均失败 | STOP，报告失败原因 |
| MEMORY.md 格式损坏 | STOP，要求用户手动修复或备份 |

**Escalation 格式**：
```markdown
STATUS: BLOCKED
REASON: [问题描述]
AFFECTED_FILE: [受影响的文件]
ATTEMPTED: [尝试了什么]
RECOMMENDATION: [建议用户做什么]
```
